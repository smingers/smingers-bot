### Key Technical Points from the Article:

**The Core Problem with Google Trends Data:**
- Google Trends normalizes each query window independently — the highest value in any given window is set to 100, with everything else scaled relative to that peak
- This means naively stitching together multiple time windows produces inconsistent, incomparable data

**The Proposed Solution — Rolling Windows with Overlap:**
- Pull 90-day windows, advancing 60 days each time, creating a **30-day overlap** between consecutive windows
- Use the overlap period to calculate a **scaling factor** between batches
- Chain scaling factors back to Batch 1 to normalize all data to a single consistent scale

**Practical Challenges Noted:**
- Rate limiting from the Google Trends API requires retry logic with sleep delays
- Real-world data collection code is often "hacked together" rather than elegant
- Intermediate results should be saved to CSV to avoid data loss from interruptions

**Result:** The scaled data produces a smoother, continuous trend line with peaks aligning to the raw Google Trends data, but on a consistent scale across the full time period.

## Summary: FAQ About Google Trends Data (Support.google)

This article explains how Google Trends works, covering data collection, normalization, and interpretation.

### Data Collection & Sampling
- Google Trends uses a **sample** (not the entirety) of actual Google searches, which is considered sufficient given billions of daily searches.
- Data is **anonymized, categorized, and aggregated**.
- The sample is designed to be representative and can reflect insights within minutes of real-world events.

### Normalization
- Data points are divided by total searches for the given geography and time range (to avoid high-volume regions always ranking highest).
- Results are then **scaled 0–100** based on a topic's proportion to all searches.

### What's Filtered Out
- Searches by very few people (appear as "0")
- Duplicate searches from the same person in a short period
- Queries with special characters
- Searches made by Google's own products/services (including AI Mode and AI Overviews)
- Note: Some irregular/automated searches **may be retained** as a security measure.

### Important Caveats
- Google Trends is **not a scientific poll** and should be treated as one data point among many.
- A spike reflects search interest, not necessarily popularity or "winning."

### Time Zone Usage (Relevant to Resolution)
- For **time ranges of 30 days or longer** (daily/weekly/monthly granularity): data uses **UTC**.
- For **time ranges of 7 days or shorter** (hourly granularity): data uses the **user's local time zone**.
